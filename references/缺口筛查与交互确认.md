# 缺口筛查与交互确认（CP5）

> 用途：推荐筛查第三部分“周期性筛查管理”。由 LLM 读取知识库和时间线分析；脚本仅做 PUA 校验，不计算缺口、不代替去重。

## 一、执行时点

CP5 在健康总结结构化和快照式癌症后验完成后执行，与 CP2 风险因素问诊分离。

先完成：

1. A：中等风险及以上癌症对应的筛查；
2. B：健康总结中的癌症外疾病和异常指标复查；
3. C：按年龄、性别、指南周期和档案时间线识别的周期性筛查候选。

优先级为 `A > B > C`。LLM 在生成问题前完成去重；已由 A/B 推荐的检查从 C 删除，不再询问。

## 二、LLM 必读输入

- `demographics.json`（年龄、性别——查 schedule 的键）
- `snapshot_risk.json`
- `health_summary_structured_summary.json`
- 当前 `content.md`、`refined.md` 和结构化检查产物
- 历史 `screening_test_timeline.json`（存在时）
- **`screening_general/json/periodic_screening_schedule.json`（v2.0.2 核心知识——结构化年龄/性别→筛查起始+间隔表，覆盖癌+非癌）**
- `screening_personalized/md/` 中与 A/B 相关的指南（用于 A/B 推荐方法/周期细节）

## 三、缺口判断

### 3.1 查表确定应筛项目

LLM 读 `periodic_screening_schedule.json`，按患者 **age + gender** 筛选 `start_age ≤ age ≤ stop_age` 且 `gender` 匹配的所有 items。每个 item 给出：

- `name`：筛查项目名
- `method`：检查方法
- `interval_years`：**推荐间隔（几年一次）——判 overdue 的基准**
- `high_risk_only`：true 的项目仅高危人群适用（须先确认危险因素，如吸烟≥20包年/HBV/Hp感染/家族史等）；非高危人群跳过。

### 3.2 对照档案判 gap_status

对每个应筛项目，LLM 全文核查 `content.md`、`refined.md` 和历史 `screening_test_timeline.json` 中的检查记录（用项目名/方法名/别名搜索）：

- **`never_recorded`**：当前和历史档案均无相关检查信息。
- **`overdue`**：有可确认日期的记录，但距今已超过 `interval_years` 年。
  - 判定方式：报告检查日期（或当前日期） − 上次检查日期 > `interval_years`
  - 例：结肠镜 `interval_years=10`，上次 2014-06-01，当前 2026-06-26 → 间隔 12 年 > 10 年 → overdue。
- **`unverifiable_date`**：有检查信息但日期不完整，无法判断是否仍在周期内。

判定前必须全文反向核验。不能把”健康总结未展示”视为”从未做过”。例如判断结肠镜缺口前，检查肠镜、胃肠镜、结肠镜、息肉、病理、切除、Boston 评分等表达。

## 四、草案和去重

写 `screening_recommendations_draft.json`：

- `cancer_risk`：A；
- `other_abnormalities`：B；
- `periodic_candidates`：去重后的 C；
- `dedup_audit`：记录从低优先级类别删除的检查及原因。

每条建议包含稳定 `dedup_key`、知识库文件、知识库字面证据。周期候选还需记录已核查的时间线来源和原文证据。

脚本遇到重复项时只阻断并报错，不自动删除。

## 五、独立两步交互

按 `periodic_candidates` 写 `screening_gap_questionnaire.json`，每项独立询问：

1. 「根据指南建议的周期，您在该周期内做过 [检查项目] 吗？」
   - `done`
   - `not_done`
   - `unknown`
2. 仅 `done` 时追问结果：
   - `normal`
   - `abnormal`
   - `unknown`

问题逐项新启动，不并入 CP2。答案保存为 `<out>/screening_gap_answers.json`，运行时通过 `--screening-gap-answers` 传入。

## 六、回答后更新

LLM 写 `screening_recommendations_final.json`：

| 回答 | 最终处理 |
|---|---|
| 未做过 | 纳入 C |
| 不清楚是否做过 | 纳入 C |
| 做过且异常 | 纳入 C，改为异常后复查建议 |
| 做过但结果不清楚 | 纳入 C |
| 做过且正常 | 排除，写入 `excluded_done_normal` |

LLM 再次检查 A/B/C 的 `dedup_key`。C 只能进入 `timeline_tiers.maintain`，且 `dedup_key` 必须与 final 一致。

## 七、PUA 边界

- LLM 负责分析、周期判断、去重和推荐。
- 脚本只校验阶段、schema、字面证据、问答完整性、回答与 final 的一致性、final 与报告 artifact 的一致性。
- 不得从报告推断用户答案。
- 无周期候选时仍写空问卷和空 C，证明 CP5 已执行。
- 脚本不得自动生成、删除或改写医学推荐。
